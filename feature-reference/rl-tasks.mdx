> stub: RL Task is a workspace-scoped, optionally-environment-linked entity that packages a runnable RL plugin. Cyberwave generates the **SceneCfg** from a linked environment (the same MuJoCo scene used everywhere else in the platform) and you bring the rest of the MDP / RL configuration as an arbitrary tree of source files. Exporting an RL task produces a ZIP that drops directly into `cyberwave-rl/tasks/<task_id>/` and is discovered by the existing registry — `make run_cfg` then runs it like any other task.

## Why a separate entity?

> stub: We deliberately keep RL tasks independent of `Environment`. Deleting an environment **detaches** linked RL tasks (`environment=None`) instead of cascading, so a task can keep its source tree even when the world it was originally trained against is gone. This matches how researchers actually iterate: the SceneCfg is a moving target while the reward, action, and observation specs are the long-lived artefact.

## Lifecycle

> stub:
> 1. **Create** an RL task from `/rl-tasks` or from the **File → Export → Create RL task from this environment** menu in the environment editor.
> 2. **Import** an existing task ZIP (e.g. an in-tree task from `cyberwave-rl/tasks/`) to seed the source tree, or write the files directly in the editor.
> 3. **Configure** entrypoint paths (`env_cfg_path` / `env_cfg_factory`, `rl_cfg_path` / `rl_cfg_factory`, optional `registry_path`) and pick where the generated SceneCfg should land via `scene_cfg_path`.
> 4. **Regenerate SceneCfg** to write the Cyberwave-owned SceneCfg module from the linked environment. Cyberwave never enforces a particular `from … import …` statement; the import path is yours to wire.
> 5. **Export ZIP** and drop the folder into `cyberwave-rl/tasks/`. Run with `make run_cfg <task_id>` exactly like the in-tree examples.
> 6. **Round-trip**: edit locally, then re-import the same folder back into Cyberwave to keep the editor in sync.

## What's in the exported ZIP?

> stub: A minimal Python package with:
>
> - `__init__.py` — generated header that imports your registry (or the default one).
> - The user-owned tree exactly as you authored it — arbitrary nesting, free file structure.
> - The Cyberwave-owned SceneCfg module at `scene_cfg_path` (regenerated from the linked environment), the sibling `scene_bindings.py` with semantic role/twin/sensor aliases that user code consumes instead of hardcoding entity names, the strict `cyberwave_task_spec.py` (canonical Python interchange for scene entities), and a `cyberwave_export.json` manifest.
> - The MuJoCo scene assets bundled from the same `export_mujoco_zip_cloud` pipeline used elsewhere: one merged `models/mujoco_scene.xml`, one **per-entity** `models/<name>.xml` carved out of it at build time (so each `EntityCfg.spec_fn` can `MjSpec.from_file` its own slice), and the shared `models/assets/` meshes.
> - A fallback `registry.py` wiring `env_cfg` / `rl_cfg` factories into `register_cyberwave_task`, used only when you don't ship your own registry path.

## Compatibility today

> stub: The exported plugin targets one of two runtimes selected on the RL task **Runtime** tab:
>
> - **`cyberwave-rl`** — the existing Docker stack. `make start` brings up the runner; `make run_cfg arg_overrides="--task=<id>"` picks the task by id. Real training expects a CUDA GPU.
> - **`cyberwave-rl-experimental`** — plain Python install (no Docker). Useful for CPU smoke tests via `python -m train_from_cfg --device=cpu --num_envs=2 --max_steps=10` and full GPU training with the same defaults as `cyberwave-rl`.
>
> Both runtimes consume the same exported plugin contract. The exporter records the choice (plus optional version pins for `mjlab`, the chosen runtime, and `skrl`) under `cyberwave_export.json → runtime`, and ships a `cyberwave_runtime.py` adapter so the same folder is portable across runtimes. Re-importing the folder back into Cyberwave keeps the editor authoritative.

## Runtime target, accelerator & bundle version

> stub: Open the **Runtime** tab on any RL task to:
>
> 1. Pick a `runtime_target` (`cyberwave-rl-experimental` for local Python, or `cyberwave-rl` for the dockerized stack).
> 2. For the dockerized path, pick an `runtime_accelerator` (`gpu` is the default; `cpu` is the smoke-friendly opt-in).
> 3. Pin the single `runtime_bundle` version (default `latest`). The bundle ships `mjlab` + the chosen runtime + `skrl` together, so callers only ever pin one version.
> 4. Configure the entrypoints — `scene_cfg_path`, `env_cfg_path` / `env_cfg_factory`, `rl_cfg_path` / `rl_cfg_factory`, optional `registry_path` — inline as a subsection.
> 5. Copy the matching **Train locally** snippet for the selected backend + accelerator.
>
> The runtime contract (target + accelerator + versions + policy interface) is persisted on the `RLTask` and forwarded to checkpoints / inference dispatch with a `task → checkpoint → launch` merge order.

## Duplicate an RL task

> stub: The RL task editor's header and the cards on `/rl-tasks` both expose a **Duplicate** action. It clones the task definition (metadata, runtime contract, source files, scene entities) into a new row in the same workspace under `Copy of <name>`. Trained checkpoints and inference runs are **not** cloned — the duplicate is a fresh task that hasn't trained yet. The same operation is exposed as `POST /api/v1/rl-tasks/{uuid}/clone` and as `RLTaskClient.clone_task(...)` in the Python SDK.

## Inference (preview)

> stub: A separate **checkpoint → inference** path is exposed alongside the export flow. It deliberately **does not run MuJoCo** — the worker loops Cyberwave joint state through the policy and publishes commands via MQTT.
>
> - `POST /api/v1/rl-tasks/{uuid}/checkpoints` — attach a trained policy checkpoint (weights URL + runtime target + versions + policy IO interface). `GET` / `DELETE` are also available under the same prefix.
> - `POST /api/v1/rl-tasks/{uuid}/checkpoints/upload` — multipart upload that materialises both an `Attachment` and the matching `RLTaskCheckpoint` row in one call (runtime/interface fields inherit from the parent RL task).
> - `POST /api/v1/rl-tasks/{uuid}/inference` — launch an inference workload for a target `twin_uuid`. The body accepts `checkpoint_uuid` (preferred), plus optional `mode`, `max_steps`, and `control_rate_hz` overrides. The backend dispatches a `cyberwave-rl-task-inference` cloud-node workload with the merged runtime metadata.
> - `GET /api/v1/rl-tasks/{uuid}/inference` — list past dispatches with their resolved runtime target/versions and linked workload.
>
> Frontend hooks (`useRLTaskCheckpoints`, `useCreateRLTaskCheckpoint`, `useUploadRLTaskCheckpoint`, `useRLTaskInferenceRuns`, `useLaunchRLTaskInference`) wrap the same endpoints for the dashboard.

## Policies — upload trained weights and play locally

> stub: The **Policies** tab on the RL task editor is the post-training counterpart to the Runtime tab's "Train locally" snippets. After training with `cyberwave-rl`, upload your `best_agent.pt` back to the task with the **Upload policy** card (max 512 MB, `.pt` / `.pth` / `.zip` accepted); the backend registers an `RLTaskCheckpoint` row whose ACL inherits from the RL task. The **Registered policies** list shows attachment-backed uploads ("Uploaded" badge) and external `weights_url` rows ("URL" badge) side by side and exposes `Download` / `Delete` actions. Selecting a checkpoint reveals a copy-paste **Use in MuJoCo locally** snippet that uses the SDK to download the checkpoint, stage it into the `logs/<task>/uploaded/checkpoints/best_agent.pt` layout that `cyberwave-rl/source/play.py` expects, and run `PlayConfig`. The Python SDK exposes the same workflow as `RLTaskClient.upload_checkpoint(...)`, `RLTaskClient.download_checkpoint(...)`, and `RLTaskClient.prepare_checkpoint_for_play(...)`; `cyberwave-rl`'s `play()` also auto-stages any loose checkpoint passed via `--checkpoint` so iterating on one-off files no longer requires manual directory juggling.

## API surface (preview)

> stub:
>
> - `GET/POST /api/v1/rl-tasks` — list / create.
> - `GET/PUT/DELETE /api/v1/rl-tasks/{uuid}` — read, update, soft-delete (independent of the linked environment).
> - `POST /api/v1/rl-tasks/{uuid}/clone` — duplicate the task definition into a new row (metadata + runtime contract + source files + scene entities; checkpoints and inference runs are not cloned).
> - `GET /api/v1/rl-tasks/{uuid}/source-files`, `GET .../source-files/content?path=…`, `PUT .../source-files`, `DELETE .../source-files?path=…` — manage the user-owned tree. Server-side validation rejects unsafe paths and Python files that fail to compile.
> - `POST /api/v1/rl-tasks/{uuid}/regenerate-scene-cfg` — write the SceneCfg at the configured path.
> - `GET /api/v1/rl-tasks/{uuid}/export.zip` — download the runnable plugin ZIP.
> - `POST /api/v1/rl-tasks/import.zip` and `POST /api/v1/rl-tasks/{uuid}/import.zip` — import a ZIP as a new task or refresh an existing one (Cyberwave-owned generated files are skipped; if the ZIP contains `cyberwave_task_spec.py`, the response surfaces an explicit warning because scene entities are **not** auto-restored).
> - `GET /api/v1/rl-tasks/{uuid}/task-spec.py` and `PUT /api/v1/rl-tasks/{uuid}/task-spec.py` — read or apply the strict scene-entity task spec explicitly. Use this when you actually want to re-apply a `cyberwave_task_spec.py` body to the live composer state.
> - `GET/POST /api/v1/rl-tasks/{uuid}/checkpoints` and `GET/DELETE /api/v1/rl-tasks/{uuid}/checkpoints/{checkpoint_uuid}` — manage trained policy checkpoints (weights URL + runtime metadata + policy interface).
> - `GET/POST /api/v1/rl-tasks/{uuid}/inference` — launch and list backend-dispatched policy inference workloads (no MuJoCo loop).

## Export / generated-file rules

> stub: The export ZIP always carries fresh bytes for **Cyberwave-owned** generated files (`scene_cfg.py` at the configured path, `scene_bindings.py`, and `cyberwave_task_spec.py`). Stored `RLTaskSourceFile` rows for those paths are skipped during export and the renderer writes current DB state instead — so re-running the composer (or hitting `POST /regenerate-scene-cfg`) is enough to refresh the artefact. The same `render_task_spec_for_rl_task` helper is used by the API GET, the regenerate endpoint, and the export ZIP, so all three produce byte-identical output for the same scene-entity rows. Body-resolution failures during MJCF splitting surface as **HTTP 400** with the failing entity name and the selectors that were tried, instead of a generic 500; check the twin selectors and re-export.

> stub: `scene_bindings.py` is the stable interface between Cyberwave-owned scene metadata and user-owned training code. It exposes role aliases (`ROBOT_ENTITY`, `OBJECT_ENTITY`, `SUPPORT_ENTITY`), twin namespace prefixes, namespaced joint-position regex keys, actuator target regexes, camera attach paths, contact sensor names, and a `body_pattern(entity, suffix)` helper. Consume those constants from `env_cfg.py` so renaming a twin or moving a camera in the composer regenerates the bindings without touching MDP code.

> stub: ZIP **import** is intentionally **document-only** for the generated paths: `scene_cfg.py`, `scene_bindings.py`, and `cyberwave_task_spec.py` are skipped on import and the response carries an explicit warning. Scene entities are **not** auto-restored from a bundled `cyberwave_task_spec.py` — re-apply it through `PUT /api/v1/rl-tasks/{uuid}/task-spec.py` or compose the entities again in the editor.

## Orchestration tabs (Actions / Observations / RL config)

> stub: Three editor tabs persist structured metadata for the MDP I/O surface and the RL training contract, alongside the existing **Scene entities** tab. The persisted JSON is exported into `scene_bindings.py` (as `ACTION_SPEC`, `OBSERVATION_SPEC`, `RL_CONFIG_SPEC`, plus `ACTION_NAMES` / `OBSERVATION_GROUPS` / `POLICY_OBSERVATION_NAMES`) and into the strict `cyberwave_task_spec.py` interchange. User-owned `env_cfg.py` / `rl_cfg.py` consume those constants; rewards, resets, terminations, and metrics remain user-authored.

> stub: **Actions** are authored via four control-mode presets: **Position delta** (`action = 0` holds the current pose, maps to `JointDeltaPositionActionCfg`), **Position target** (absolute target, anchored at joint home), **Velocity**, and **Effort / torque**. Each preset seeds canonical defaults (`scale`, `offset`, `use_default_offset`). The Scene tab is where you turn on **simulation actuators** for the entity (which `Xml*ActuatorCfg` wrapper class drives each joint); the Actions tab is where you pick the **policy-controlled subset** of those enabled actuators. Each action row renders a schema-backed joint table mirroring the Scene tab columns (Joint, Control type, Position min/max, Max velocity, Max torque/force, KP, KD, Selected) with **Select all** / **Clear** controls — passive joints, spring-only actuators, and joints driven by an incompatible wrapper class appear in the table greyed out with a hint so users can fix the asset patch or Scene actuator group. The persisted spec stores concrete joint names in `target_names_expr`; a collapsed **Advanced — custom regex** disclosure lets imported / legacy specs keep regex patterns. Presets that are incompatible with the chosen entity (e.g. `Effort / torque` against an `xml_position` actuator) are hidden, not shown disabled. `position_delta` / `position` require an `xml_position` actuator; `velocity` accepts `xml_velocity` or `xml_position`; `effort` requires `xml_motor`. The Save button stays disabled until every row passes client-side validation.

> stub: **One action per joint.** Every `(entity, joint)` pair can be assigned to at most one action term across the whole spec — the policy emits a single value per actuated DOF, so two terms claiming the same joint would silently stack or overwrite each other. The Actions tab enforces this in three places: joints already controlled by another row render disabled with an "owned by …" pill, **Select all** only picks unclaimed joints (existing conflicts stay visible so they can be cleared, not lost silently), and `PUT /api/v1/rl-tasks/{uuid}/actions` rejects the payload with a clear error naming both action names and a sample of overlapping joints. The validator resolves `target_names_expr` against the entity's schema joints with the same prefix-tolerant matcher the orchestration hints use, so an action driven by a concrete joint list collides with a sibling action whose regex resolves to the same joint. To split a robot across two action terms (e.g. arm + gripper), give each term a disjoint subset of joints.

> stub: **Scale, offset, and baseline_delta.** Every action row exposes its formula directly under the control-mode picker, mirroring the Isaac Lab / mjlab `JointPositionActionCfg` / `JointVelocityActionCfg` / `JointEffortActionCfg` math one-to-one. `position`, `velocity`, and `effort` use `target = action × scale + offset` — exactly the formula mjlab implements in `BaseAction.process_actions`. A single `Use default joint offset` checkbox mirrors mjlab's `use_default_offset` boolean: when on, mjlab replaces `offset` with each joint's `default_joint_pos` (or `default_joint_vel`) at runtime so action = 0 returns the joint to its home pose; turning it off lets the user type a literal numeric bias (scalar or per-joint map). `position_delta` is Cyberwave's own preset and uses `target = q_now + baseline_delta + action × scale`; `baseline_delta` is a separate explicit field (it replaced the confusing legacy "Offset reference" / "Use default joint offset" pair for delta actions) that bakes a constant per-step delta into the target so action-zero can drift the joints toward a goal pose. Every numeric field (`scale`, `offset`, `baseline_delta`) carries a **Per joint** toggle that switches the value between a uniform scalar and a `{joint_name: value}` mapping; flipping the toggle on opens an inline column in the joint table for editing per-joint values. The backend validator rejects per-joint mapping keys that aren't in the action's selected joints, so typos surface at save time instead of dropping silently at runtime. Legacy symbolic offsets (`"default"` / `"zero"` / `"none"`) keep round-tripping unchanged.

> stub: **Observations** group terms (`policy`, `critic`, …) and support the catalog: `joint_position`, `joint_velocity`, `joint_effort`, `link_position`, `link_pose` (xyz + quaternion), `link_velocity` (xyz linear + rx/ry/rz angular twist), `camera` (depth/color from a sensor configured on a scene entity), `previous_action`, and `custom` (user `module.symbol` + literal kwargs). Link terms pick bodies from a scene-derived checkbox list (no more typing names by hand) and expose **Components** toggles so users only export the axes the policy needs — `link_position` picks among `x`/`y`/`z`, `link_velocity` adds `rx`/`ry`/`rz` for the angular twist, and `link_pose` exposes `x`/`y`/`z` plus a single `quat` token that keeps the full `(qw, qx, qy, qz)` tuple grouped so the runtime never sees a half rotation. The tab toggles between **Shared policy/critic observations** (default — one group authored, the runtime aliases the missing `critic` group onto `policy` so both heads see the same tensor) and **Separate policy and critic groups** (author each group independently). Observation rows show the term type as a compact chip next to the editable name (the type is fixed after creation — remove + re-add to change it). Joint observation terms reuse the shared **Joint target picker** — observations may include both actuated and passive joints (unlike actions), with "All actuated", "All joints", "Passive joints", and "Pick specific joints" exposed directly; "Custom regex" sits behind a small **Advanced** disclosure to keep the common path uncluttered.

> stub: **RL config** persists the skrl backend + algorithm (PPO or SAC) + a structured **Network builder** (rlmodule library) + trainer parameters (timesteps, intervals, seed) + the algorithm-config knobs (learning rate, discount factor, ratio clip, batch size, …). Switching algorithm seeds matching defaults; saving normalises and round-trips through the API. **Experiment paths** (`experiment.name` / `experiment.directory`) are derived from the RL task slug by default and shown as read-only helper text; toggle **Override experiment paths** to write custom values.

> stub: **Network builder** is graph-first. The RL config tab picks a *topology* and renders the network as a DAG of typed nodes on an inline canvas; there is no separate "MLP form vs graph" editor any more. `shared` means "use the same architecture for both heads" — for PPO that yields a true shared-trunk model (`SharedRLModelCfg`), and for SAC it instantiates two independent modules (policy + Q) with the same block spec because SAC heads can't share weights. `separated` lets the user author one graph per role (`policy` / `value` for PPO, `policy` / `q` for SAC). Top-level **presets** are graph templates — e.g. *PPO – shared MLP* / *SAC – separated MLP* — that materialise the equivalent observation → trunk → head topology directly on the canvas. PPO algorithm-config fields are ordered by training impact: the primary row shows `rollouts`, `discount_factor`, `learning_rate`, and `entropy_loss_scale`, with the rest collapsed into an advanced section. Legacy `{kind, hidden_units, activation}` and `architecture: "mlp"` / `"residual_mlp"` / `"custom"` specs are transparently **promoted** to a graph on load: hydration synthesizes equivalent nodes so the inline canvas just renders, and the `blocks` field stays populated alongside `graphs` for backend / runtime back-compat. At runtime, `from runners.skrl.network_builder import build_skrl_model_cfg` maps the persisted spec onto `rlmodule.MlpCfg` / `ResidualMlpCfg` / `SharedRLModelCfg` / `SeparatedRLModelCfg` for the skrl runner.

> stub: **Inline graph canvas (`architecture: "graph"`)** mounts directly on the RL config tab — one canvas per role for the active topology. The canvas has a subtle dot-grid background and a fixed height; drag the empty background to **pan** around the workspace, drag any node to **reposition** it (positions are persisted in `graphs[role].layout` on pointerup), and click an edge to delete it. Clicking a node opens an **inline inspector below the canvas** showing the outputs picker (`features_node_id` / `head_node_id`) plus the per-node editor — no sidebar dance required. The canvas starts in *compact mode* with the palette collapsed and the inspector hidden until a node is selected; the small **palette** / **inspector** toggle buttons in the canvas header let you pop them open or shut them down manually. There is no passive-aggressive “Graph looks valid.” copy in the header — warnings only render when the validator has something real to flag. An **Open graph builder** button on each role pops the same graph into a 92vh/96vw fullscreen dialog with the palette + canvas + sidebar inspector all expanded for serious authoring. Closing the dialog keeps the draft state; the existing **Save RL config** flow persists everything together. The palette covers inputs (`observation_input`), dense (`mlp`, `residual_mlp`), transforms (`activation`, `layer_norm`), recurrent (`rnn`, `gru`, `lstm`, one per role for v1), vision (`cnn_backbone` with NatureCNN / IMPALA presets), merge (`concat`, `residual_add`), and a role/algorithm-aware **Heads** section that only surfaces `actor_head` + `value_head` for PPO graphs and `actor_head` + `q_head` for SAC graphs. `action_input` and `projection` remain part of the persisted node catalogue (so old graphs still load) but are hidden from the normal palette to keep the surface focused.

> stub: **Observation routing and tensor metadata.** `observation_input` nodes pick a group and a subset of term names from the **Observations** tab. The inspector shows each term as a chip annotated with read-only tensor metadata — an `image-like` / `flat` badge and a short kind label (`joint pos`, `link pose`, `depth camera`, …) — derived from the term type instead of a user-edited modality dropdown (all observations are floats; what actually matters is whether the slice can be reshaped into `[H, W, C]` for a CNN). Selecting only image-like terms upgrades the node to an image-routed input and surfaces an optional `[H, W, C]` reshape override for cases where the camera sensor width/height isn't carried by the orchestration hints. Frontend validation rejects flat terms feeding a `cnn_backbone`, warns on mixed image/flat selections, and the backend mirrors the same rules on save. **Heads by role** are enforced both in the palette and on save: `q_head` is rejected in PPO graphs (even when not the configured terminal), `value_head` is rejected in SAC graphs, and the per-role policy/value/q split allows exactly one head type. **Gaussian log-std controls** (`initial_log_std`, `min_log_std`, `max_log_std`) live inside the actor head inspector in graph mode — they only apply to the stochastic Gaussian policy, never to deterministic value or Q heads — but persist on `network.output` for backend / runtime compatibility.

> stub: Endpoints: `GET/PUT /api/v1/rl-tasks/{uuid}/actions`, `GET/PUT /api/v1/rl-tasks/{uuid}/observations`, `GET/PUT /api/v1/rl-tasks/{uuid}/rl-config`, plus `GET /api/v1/rl-tasks/{uuid}/orchestration-hints` which returns scene-derived dropdown data (articulation entities, actuator groups + types per entity, declared joint/body names, **actuated vs passive** joint lists for the picker, configured camera and contact sensors). The `RLTask` response itself surfaces the three specs as `action_spec` / `observation_spec` / `rl_config_spec` JSON fields.

> stub: **Runtime spec adapters.** Three execution-only modules in `cyberwave-rl` consume the persisted specs so user-owned files only have to provide rewards, resets, terminations, commands, and custom observation functions:
>
> - `from task_export.action_specs import make_action_cfgs` — `ACTION_SPEC` → `{action_name: ActionTermCfg}` (`JointDeltaPositionActionCfg` / `JointPositionActionCfg` / `JointVelocityActionCfg` / `JointEffortActionCfg`). The `position_delta` runtime class lives in `cyberwave-rl/source/mdp/actions/joint_delta_position_action.py`.
> - `from task_export.observation_specs import make_observation_groups` — `OBSERVATION_SPEC` → `{group_name: ObservationGroupCfg}`. Joint / link / camera / `previous_action` terms route through small per-category resolvers (the user wires `mdp.joint_pos_rel`, `mdp.last_action`, etc. once); `custom` terms are looked up in a `{name: ObservationTermCfg}` registry the user controls.
> - `from task_export.rl_config_specs import make_rl_cfg_from_spec` — `RL_CONFIG_SPEC` → `ReinforcementLearningCfg` (PPO/skrl + rlmodule-built network + experiment + trainer wiring).
>
> Authoring (presets, joint picker, term catalogue, network/PPO defaults) stays in the Cyberwave editor/backend.

> stub: **Cyberwave-owned vs user-owned.** The editor authors Scene entities + Actions + Observations + RL config + network builder, plus all generated artefacts (`scene_cfg.py`, `scene_bindings.py`, `cyberwave_task_spec.py`, the export ZIP). Task-local Python — reward functions, reset / termination logic, metrics, command shaping, and custom observation functions (e.g. wrist-depth preprocessing, object-relative features) — remains user-owned under `task_sources/`. The Python SDK mirrors the editor surface with `RLTaskClient.get_action_spec` / `set_action_spec`, `get_observation_spec` / `set_observation_spec`, `get_rl_config_spec` / `set_rl_config_spec`, and `get_orchestration_hints`, plus low-level builders (`make_action_spec` / `make_action_term` / `make_observation_spec` / `make_observation_term` / `make_rl_config_spec`) **and** one-liner editor presets (`make_position_delta_action`, `make_joint_observation`, `make_previous_action_observation`, `make_camera_observation`, `make_custom_observation`, `make_default_ppo_rlmodule_config`) so setup scripts stay declarative.
