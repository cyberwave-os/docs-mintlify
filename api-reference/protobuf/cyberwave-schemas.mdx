Protobuf sources live under `common/protobuf`. MQTT `.proto` files document **broker topics** near the top; those paragraphs are surfaced in Mintlify `<Note>` blocks, with **`Payload`** message names listed below.

You can usually **predict the MQTT path from the filepath** (`v0/` layout): omit the **`mqtt/`** directory segment while keeping **`cyberwave/<scope>/…`**, prepend `{prefix}`, and insert the UUID placeholder **after the first path segment** — for example `…/mqtt/twin/position.proto` → `{prefix}cyberwave/twin/{twin_uuid}/position`, `…/mqtt/environment/telemetry_status.proto` → `{prefix}cyberwave/environment/{environment_uuid}/telemetry-status`. Some payloads need extra suffix variables (`{update_type}`), and a few protos list **multiple** topics (`webrtc`, workflow execution). Prefer the authoritative `Topic:` / `Topics:` header in source; **path-derived hints** appear in a `<Note>` only when that header omits topic lines (see `scripts/enrich-md-proto-leads.py`).

Refresh after schema edits:

```bash
cd common/protobuf && make mdx-docs
```

Running `buf generate` alone leaves `<!-- proto-lead:… -->` placeholders in this file until the Make target runs `scripts/enrich-md-proto-leads.py`.

See also the topic catalogue & JSON examples at [MQTT API reference](/api-reference/mqtt/main).

<Note>

## cyberwave/mqtt/joint/update.proto

<Note>

Topic: {prefix}cyberwave/joint/{twin_uuid}/update

Per-joint or multi-joint state updates published by the main SDK MQTT client (CyberwaveMQTTClient.update_joint_state / update_joints_state).

Two wire formats co-exist:

1. Single-joint format (JointSingleUpdatePayload): Published per-joint by update_joint_state(). Contains a `joint_name` field and a `joint_state` sub-object with optional position/velocity/effort. Backend rate-limits per (twin_uuid, joint_name) pair.

2. Flat multi-joint format (JointFlatUpdatePayload): Published by update_joints_state() with positions-only. Joint names are top-level keys and values are floats. Lightweight and low overhead.

3. Aggregated multi-joint format (JointAggregatedUpdatePayload): Published by update_joints_state() with velocities, efforts, timestamps, or inference metadata. Fully structured; used for telemetry-rich pipelines.

Note: The topic scope is `joint` (not `twin`) to allow per-joint rate limiting at the broker level and to keep joint streams separate from other twin state channels.

</Note>

## cyberwave/mqtt/twin/edge_health.proto

<Note>

Topic: {prefix}cyberwave/twin/{twin_uuid}/edge_health

Periodic liveness heartbeat published by edge devices approximately every 5 seconds. Two separate systems consume it:

1. Vector — persists the full payload to app_twintelemetry for analytics and drives the twin-level "live" indicator.

2. Django MQTT consumer — extracts the edge fingerprint from the associated Twin.metadata, then debounces writes to Edge.last_heartbeat (~30 s window) so that Edge.is_online remains in sync.

NOTE: This topic is deliberately excluded from cloud MQTT workflow dispatch because its high frequency (~5 s per twin) would create a continuous Celery task fan-out.
