## What Are Online Controllers?

Online controllers (`code`, `random`, `rltask`) each run in their **own
controller-host cloud node** — just like VLA controllers (`smolvla` / `pi05`).
The backend dispatches them over MQTT/WebRTC to a remote **plant**
(`cyberwave-sim` or a live robot). The plant only steps physics and publishes
state; it never runs the controller in-loop.

| | Plant (`cyberwave-sim`) | Controller host (cloud node) |
|---|---|---|
| **Role** | Pure physics + state publish | Runs the `control_fn` |
| **Communicates via** | MQTT joint state / commands, WebRTC frames | MQTT joint commands, WebRTC frames |
| **Examples** | MuJoCo / mjlab worker | `cyberwave-code-controller`, `cyberwave-random-controller`, `cyberwave-rl-controller` |

---

## How Controllers Are Attached

The **twin-assigned controller policy** is the single source of truth. When you
assign a controller to a robot twin and start a simulation, the backend
dispatches it to the matching controller host:

1. **Assign** a controller policy to your robot twin (via the UI controller panel or API).
2. **Start simulation** — the backend resolves assigned policies and dispatches every online controller as an external `inference` workload.
3. **Routing** is by explicit policy type + structured `controller_family` (never keyword matching):

| Controller | Routing | Controller-host profile |
|---|---|---|
| `code` | `code` policy | `cyberwave-code-controller` |
| `random` | `code` policy, `controller_family="random_policy"` | `cyberwave-random-controller` |
| `rltask` | `mlmodel` policy, `controller_family="rl_task_policy"` | `cyberwave-rl-controller` |

`teleop`, `navigation`, `motion`, and `vla` use their existing command-triggered / cloud paths.

## Runtime Interfaces

Every online controller interacts with the plant through a sim interface:

| Interface | Class | Used by |
|---|---|---|
| MuJoCo-shaped view | `MujocoInterface` | `code`, `code+weights` |
| SimulationLike protocol | `MjWarpInterface` | `rltask` (RL policies) |

### Function mode (default)

```python
def control(sim, sim_time, artifacts=None):
    sim.data.ctrl[:] = ...
```

### Class mode (`entrypoint_type="class"`)

```python
class Controller:
    def __init__(self, sim, artifacts=None):
        self.sim = sim
    def __call__(self, sim_time):
        self.sim.data.ctrl[:] = ...
```

Class mode instantiates the controller once with `sim` (and optional `artifacts`) and calls `__call__` each tick. Use this for stateful controllers that need to load weights or maintain internal state.

---

## Named Artifacts / Weights

Code+weights controllers can attach named binary files (`.pt`, `.onnx`, `.safetensors`). Each artifact has a **filename** (the runtime key), an optional **role** label, and a **primary** flag:

- In **function mode**, artifacts are passed as the third `artifacts` dict.
- In **class mode**, artifacts are passed to `__init__`.
- Artifacts can be added, removed, renamed, and re-ordered in the policy editor.

## Controller Compute (`cpu` / `gpu`)

Controller policies that **carry a model** (`mlmodel` / `vla`, `random`, or
`code_weights` family) accept an inference `device`:

```python
cw.controller_policies.update(uuid="<policy>", device="gpu")  # or "cpu" (default)
```

- This is **independent** of the plant's physics device (simulation backend).
- The **saved device** on the policy is the default; at deploy time you can override it per-run without changing the saved policy.
- It is rejected for policies that carry no model (e.g. a plain `code` or `teleop` policy).

## Quick Example

Assign a code controller to a twin, then start a simulation — the controller auto-runs:

```python
from cyberwave import Cyberwave

cw = Cyberwave()

# Assign a code controller to the twin
cw.twins.update(
    twin_uuid="...",
    controller_policy_uuid="<policy-with-code>",
)

# Start simulation — assigned controllers run automatically
cw.simulation.start(environment_uuid="...")
```

For explicit override (advanced), you can still pass `online_controllers` directly:

```python
cw.simulation.start(
    environment_uuid="...",
    online_controllers=[
        {
            "type": "python_artifact",
            "code": "def control(sim, sim_time):\n    sim.set_joint_targets({})",
            "target_twin_uuid": "...",
        }
    ],
)
```

## Simulation Backends

Online controllers work with both simulation backends:

- **Basic MuJoCo** — single `MjModel`/`MjData`, direct physics callbacks
- **Vectorized mjlab** — manager-based RL environment, GPU-capable

Controllers drive both backends over the same MQTT command/state wire, so they are portable.

---

## Choosing CPU or GPU

{/* stub — content pending human review */}

When you **Start Controller**, model-carrying controllers (`random`, `rltask`)
expose a compute selector:

- **CPU** — runs the controller on a cheap GCP CPU host. Default, and the only
  option for plain `code` controllers (a `control()` loop driving a sim needs no GPU).
- **GPU** — runs the controller on a Crusoe GPU host. Use for heavier policy inference.

The choice applies to that run only and does not change the saved policy.

---

## Live Mode

Because the controller host talks to its plant purely over MQTT/WebRTC, the same
`code` / `random` / `rltask` controller can target a **live robot** plant instead
of `cyberwave-sim` — no controller code changes required. Other twins continue to
use their existing paths:

- `teleop` → MQTT keyboard/joystick control
- `navigation` → command-triggered navigation
- `motion` → backend sequence execution
- `mlmodel`/`vla` → cloud inference
